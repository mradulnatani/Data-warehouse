## **Abstract**

The rapid growth of transactional data in e-commerce systems necessitates efficient storage and analytical processing mechanisms. Traditional Online Transaction Processing (OLTP) systems are not optimized for complex analytical queries, leading to the adoption of data warehousing solutions. This project presents the design and implementation of a scalable data warehouse system using a real-world Amazon e-commerce dataset. The system incorporates a star schema-based dimensional model, a Python-driven ETL pipeline for data transformation, PostgreSQL as the storage engine, and a Streamlit-based interactive dashboard for visualization. The implementation supports Online Analytical Processing (OLAP) operations such as aggregation, grouping, filtering, and time-series analysis. Experimental results demonstrate efficient query execution and meaningful analytical insights, validating the effectiveness of the proposed system.

## **I. Introduction**

The exponential increase in data generated by modern applications, particularly in e-commerce, has created the need for systems capable of efficient analytical processing. While OLTP systems are optimized for real-time transactional operations, they are not suitable for complex queries involving aggregations, historical data analysis, and multidimensional exploration.

Data warehousing addresses this limitation by providing a centralized repository optimized for analytical workloads. It separates transactional processing from analytical processing, thereby improving performance and scalability.

This project aims to design and implement a complete data warehousing solution using a real-world Amazon e-commerce dataset. The system integrates ETL processes, dimensional modeling, OLAP operations, and visualization to enable comprehensive data analysis.

---

## **II. Problem Statement**

E-commerce platforms generate large volumes of structured and semi-structured data. Analyzing this data directly from raw sources is inefficient due to:

* High query latency
* Lack of optimized schema for analytics
* Redundant and inconsistent data
* Difficulty in performing multidimensional analysis

The problem addressed in this project is to transform raw transactional data into a structured format that supports efficient analytical queries and visualization.

## **IV. Dataset Description**

The dataset used in this project is sourced from Kaggle and represents Amazon e-commerce transactions. It consists of approximately one million records and includes attributes such as:

* **User Attributes**: user_id, location, device
* **Product Attributes**: product_id, category, subcategory, brand
* **Transaction Attributes**: price, discount, final_price
* **Seller Attributes**: seller_id, seller_rating
* **Temporal Attributes**: purchase_date, shipping_time_days
* **Operational Attributes**: payment_method, delivery_status, is_returned

The dataset simulates real-world e-commerce operations and contains sufficient complexity for analytical processing.

## **VI. Data Warehouse Design**

### A. Dimensional Modeling

The system uses **dimensional modeling**, which organizes data into fact and dimension tables. This approach improves query performance and simplifies analytical queries.

---

### B. Star Schema

A **star schema** is implemented due to its simplicity and efficiency. It consists of:

* One central fact table
* Multiple dimension tables connected via foreign keys

---

### C. Fact Table: `sales_fact`

The fact table stores measurable metrics:

* price
* discount
* final_price
* shipping_time_days
* is_returned

It also contains foreign keys referencing dimension tables.

---

### D. Dimension Tables

1. **user_dim**

   * Stores user attributes
   * Uses natural key (user_id)

2. **product_dim**

   * Stores product hierarchy
   * Enables category-level analysis

3. **seller_dim**

   * Stores seller ratings

4. **time_dim**

   * Enables temporal analysis
   * Includes derived attributes (day, month, year)

5. **payment_dim**

   * Stores payment methods

6. **delivery_dim**

   * Stores delivery status

---

### E. Design Considerations

* Natural keys retained for simplicity
* Surrogate keys used where required
* Duplicate handling implemented
* Optimized for read-heavy workloads

## **VII. ETL Pipeline**

The ETL pipeline is a critical component of the system.

---

### A. Extract Phase

* Data is read from CSV using Pandas
* Ensures efficient memory handling

---

### B. Transform Phase

Transformations include:

#### 1. Data Cleaning

* Removal of null values
* String normalization (lowercase, trimming)

#### 2. Type Conversion

* IDs converted to string
* Dates converted to datetime format

#### 3. Deduplication

* Ensures unique dimension entries
* Uses subset-based filtering

#### 4. Dimension Construction

* Extract unique values for each dimension
* Generate derived attributes

#### 5. Key Mapping

* Join transformed data with dimension tables
* Assign foreign keys

---

### C. Load Phase

* Data is loaded using SQLAlchemy
* Dimension tables are populated first
* Fact table is populated after merging keys

---

## **VIII. OLAP Operations**

### A. Aggregation


---

### B. Roll-Up

Data aggregated from detailed to summary level:

* Daily → Monthly

---

### C. Drill-Down

Allows deeper analysis:

* Category → Subcategory

---

### D. Slice and Dice

Filtering data across dimensions:

* Specific category
* Specific time period

### E. Implementation
